# 项目架构

## 项目定位

该项目是一个基于 `Vue 3 + Vite + TypeScript + Pinia + Element Plus` 的 RBAC 管理后台模板。

当前目标能力：

- 单 Token 登录鉴权
- 基于角色、权限码、菜单树的访问控制
- 模块路由与后端菜单动态路由并存
- 可扩展的布局、标签页、主题配置与 Mock 数据

## 启动流程

应用入口位于 `src/main.ts`，当前启动顺序如下：

1. `createApp(App)` 创建应用实例
2. `setupRouter(app)` 注册 router
3. `setupStore(app)` 注册 Pinia
4. `useThemeStore(store).initTheme()` 初始化主题
5. `setupRouterGuards(router)` 注册全局守卫
6. `app.use(ElementPlus, { locale: zhCn })`
7. `setupDirectives(app)` 注册 `v-permission`、`v-role`
8. `app.mount("#app")`

这个顺序的目的很明确：

- 先完成 router/store 注册，再让守卫和指令可以访问 store
- 在首屏渲染前完成主题初始化，避免闪烁

## 分层结构

### 1. 应用启动层

目录与文件：

- `src/main.ts`
- `src/App.vue`

职责：

- 组装应用运行时依赖
- 控制初始化顺序
- 挂载全局 UI 能力与路由守卫

### 2. 路由层

目录：`src/router`

核心文件：

- `index.ts`：创建 router，使用 `createWebHashHistory()`
- `static-routes.ts`：定义 `Login`、`Forbidden`、`Root`、`NotFound`
- `module-routes.ts`：自动聚合 `router/modules/**/*.ts`
- `dynamic-routes.ts`：将后端 `menus` 转成 `RouteRecordRaw[]`
- `guard.ts`：负责鉴权、初始化、动态路由挂载与根路径重定向
- `modules/nested.ts`：静态模块路由示例

当前路由模型：

- 静态路由在 router 创建时注册
- 模块路由作为 `Root.children` 的初始子路由
- 动态菜单路由在首次进入受保护页面时注入到 `Root`

说明：

- 根路径 `/` 不绑定固定首页，而是由 `menuStore.resolveRootRedirectTarget()` 决定
- 页面级权限依赖 `route.meta.permission`
- 公开路由依赖 `route.meta.public`

### 3. 状态管理层

目录：`src/stores/modules`

当前 store 划分如下：

- `auth.ts`
  - 负责 `token`、`user`、`isInitialized`
  - 负责 `login()`、`fetchProfile()`、`logoutLocal()`
- `permission.ts`
  - 负责 `roles`、`permissions`
  - 负责 `hasAny()`、`hasAll()`、`hasRole()`
  - 负责动态路由挂载与移除回调管理
- `menu.ts`
  - 负责 `rawMenus`
  - 负责 `sidebarMenus`、`dynamicRoutes`、`getBreadcrumbs()`
  - 负责记录 `path -> fullPath` 映射，用于根路径重定向和面包屑回跳
- `tabs.ts`
  - 负责页签栏状态
  - 自动过滤公开页、隐藏页、403、404
- `ui.ts`
  - 负责侧边栏折叠、导航模式、页宽、灰度模式、色弱模式等 UI 偏好
  - 负责本地持久化 UI 配置
- `theme.ts`
  - 负责浅色/深色/跟随系统模式
  - 负责主题主色、CSS 变量和 Element Plus 主题变量注入

整体原则：

- `auth` 管登录态
- `permission` 管权限集合和动态路由生命周期
- `menu` 管菜单衍生视图模型
- 其余 store 管纯界面状态

### 4. 网络与接口层

目录：

- `src/api`
- `src/utils/http`
- `src/types/http.ts`

当前 HTTP 分层：

- `src/api/auth/api.ts`
  - 只描述认证接口
- `src/utils/http/client.ts`
  - 创建 Axios 实例
- `src/utils/http/interceptors.ts`
  - 注入 `Authorization`
  - 统一处理业务错误和 HTTP 错误
- `src/utils/http/request.ts`
  - 统一拼接 `baseURL`
  - 统一返回 `ApiResponse<T>`
- `src/utils/http/config.ts`
  - 定义服务地址映射，当前包含 `MOCK: "/api"`

当前接口约定：

- 所有业务接口返回统一结构：`{ code, data, message }`
- 登录相关接口使用：
  - `POST /api/auth/login`
  - `GET /api/auth/me`

### 5. 权限与指令层

目录：`src/directives`

当前自定义指令：

- `v-permission`
  - 基于权限码控制元素显示
  - 使用 `permissionStore.hasAny()`
- `v-role`
  - 基于角色控制元素显示
  - 使用 `permissionStore.hasRole()`

实现方式：

- 在 `mounted` 阶段直接移除无权限元素
- 用于按钮级和局部区域可见性控制

页面级控制仍然由路由守卫完成，指令不是主鉴权手段。

### 6. 布局与视图层

目录：

- `src/layout`
- `src/views`
- `src/components`

核心组成：

- `src/layout/AppLayout.vue`
  - 后台整体壳层
- `src/layout/components/AppSidebar.vue`
  - 基于 `menuStore.sidebarMenus` 渲染侧边栏
- `src/layout/components/AppHeader.vue`
  - 显示标题、面包屑、用户信息、设置入口
  - 在路由切换时记录 `path/fullPath` 并维护页签
- `src/layout/components/AppHeaderTags.vue`
  - 标签页导航
- `src/layout/components/AppBreadcrumb.vue`
  - 面包屑导航
- `src/components/AppRouterView.vue`
  - 统一承接嵌套路由
  - 根据 `meta.keepAlive` 控制缓存与转场

视图目录结构按业务分组：

- `views/login`
- `views/dashboard`
- `views/system`
- `views/error`

### 7. Mock 数据层

目录：`mock`

文件：

- `mock/auth.ts`
  - 基于 `vite-plugin-mock` 提供登录与用户画像接口
- `mock/data/rbac-data.ts`
  - 提供 mock 账号、角色、权限码、菜单树数据

当前内置测试账号：

- `admin / admin123`
- `auditor / auditor123`

Mock 设计原则：

- 返回结构与真实后端保持一致
- 菜单、权限、角色一次性随 `/api/auth/me` 下发

## 核心数据流

当前主链路如下：

1. 用户在登录页提交账号密码
2. `authStore.login()` 获取并持久化 token
3. 首次进入受保护页面时，守卫触发 `authStore.fetchProfile()`
4. `fetchProfile()` 将用户、角色、权限和菜单分别写入对应 store
5. `menuStore.dynamicRoutes` 从后端菜单推导出动态路由
6. `permissionStore.mountDynamicRoutes()` 将动态路由挂载到 `Root`
7. 布局组件基于 `sidebarMenus`、`getBreadcrumbs()`、`tabs` 渲染导航结构
8. 页面元素通过 `v-permission` / `v-role` 做局部可见性控制

## 当前目录关注点

如果要扩展项目，通常按下面的边界落点：

- 新增固定页面：`src/router/modules` + `src/views`
- 新增后端控制页面：后端返回 `menus`，前端补充 `src/views` 组件
- 新增按钮权限：页面模板 + 权限码配置 + 后端接口鉴权
- 新增主题能力：`src/config/theme.ts` + `theme store`
- 新增通用请求能力：`src/utils/http`

## 关键设计原则

- 单一职责：登录态、权限集合、菜单衍生、UI 状态分别由独立 store 管理
- 渐进扩展：模块路由和动态菜单路由可以并存，不互相替代
- 前后端契约优先：前端只消费 `roles / permissions / menus`，不自行推导业务权限
- 前端权限只做体验控制：真正的安全边界仍应由后端接口鉴权负责
