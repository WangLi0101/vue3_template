# 08.弹窗组件规范

## 1. 目标

本规范用于统一项目内所有弹窗组件的职责、命名、状态管理和提交流程。

适用范围不只限于 CRUD，还包括：

- 新增 / 编辑类表单弹窗
- 审批、分配、发布、重置、同步等业务操作弹窗
- 详情查看、只读展示类弹窗

目标：

- 保持所有弹窗的结构一致，降低维护成本
- 让页面层和弹窗层职责清晰
- 让新增功能时可以直接按模板落地，而不是每次重新设计写法

当前参考基准：

- `src/views/system/user/components/UserFormDialog.vue`
- `src/views/system/role/components/RoleFormDialog.vue`

## 2. 总体原则

### 2.1 页面负责打开上下文，弹窗负责内部流程

页面侧负责：

- 控制弹窗显示 / 隐藏
- 传入当前操作对象或业务上下文
- 在弹窗成功后刷新列表或刷新详情

弹窗侧负责：

- 渲染内容
- 管理内部表单 / 详情数据
- 管理校验、提交、加载状态
- 对外只暴露少量稳定事件

### 2.2 统一显隐方式

所有弹窗显隐统一使用：

```ts
const dialogVisible = defineModel<boolean>();
```

不要混用：

- `visible`
- `modelValue + update:modelValue`
- 父组件直接操作子组件内部 `ref`

### 2.3 统一生命周期钩子

所有弹窗优先统一实现：

- `open`
- `closed`

并与 `el-dialog` 事件保持一致：

```vue
<el-dialog v-model="dialogVisible" @open="open" @closed="closed" />
```

含义：

- `open`：打开后做回显、拉详情、初始化上下文
- `closed`：关闭后做重置、清理状态、避免下次脏数据

## 3. 弹窗分类规范

项目内弹窗统一分为三类，不要所有弹窗都叫 `FormDialog`。

### 3.1 `XxxFormDialog.vue`

适用场景：

- 新增
- 编辑
- 复制后新建
- 带输入项的业务提交

特征：

- 内部一定有 `el-form`
- 一般有 `rules`
- 一般存在提交按钮

推荐方法：

- `open`
- `closed`
- `getFormData`
- `add`
- `edit`
- `submitForm`

说明：

- 如果只有“新增 / 编辑”两种状态，优先使用 `isEdit: boolean`
- 不要为了两种状态额外引入 `"create" | "edit"` 字符串模式

### 3.2 `XxxActionDialog.vue`

适用场景：

- 审批
- 分配角色
- 重置密码
- 发送通知
- 上线 / 下线
- 触发同步、重试、发布等一次性业务动作

特征：

- 不一定是完整表单
- 可能只有几个字段或一段确认文案
- 核心是执行一次业务动作，而不是维护实体资料

推荐方法：

- `open`
- `closed`
- `getPayload`
- `submitAction`

说明：

- 如果需要表单校验，仍然使用 `el-form`
- 如果只是确认型弹窗，也要保留 `isSubmitting`
- 成功后统一抛出 `success`

### 3.3 `XxxDetailDialog.vue`

适用场景：

- 查看详情
- 查看日志
- 查看流程节点
- 只读预览

特征：

- 默认不提交
- 打开时拉取详情或使用传入详情
- 重点是展示，不是编辑

推荐方法：

- `open`
- `closed`
- `fetchDetail`

推荐状态：

- `loading`
- `detail`

## 4. 文件与目录规范

### 4.1 页面私有弹窗

如果弹窗只服务某个页面，放在页面私有目录：

```text
src/views/<module>/<page>/components/XxxFormDialog.vue
src/views/<module>/<page>/components/XxxActionDialog.vue
src/views/<module>/<page>/components/XxxDetailDialog.vue
```

例如：

- `src/views/system/user/components/UserFormDialog.vue`
- `src/views/system/role/components/RoleFormDialog.vue`

### 4.2 跨页面共享弹窗

如果弹窗会被多个页面复用，放到共享组件目录：

```text
src/components/business/<domain>/XxxDialog.vue
```

判定原则：

- 只在当前页面使用：放 `views/.../components`
- 两个及以上页面复用，且业务语义一致：抽到 `src/components/business/`

## 5. 命名规范

### 5.1 状态命名

通用状态优先使用以下名称：

- `dialogVisible`
- `formRef`
- `form`
- `rules`
- `loading`
- `isSubmitLoading`
- `isSubmitting`
- `detail`

说明：

- 表单提交场景优先使用 `isSubmitLoading`
- 非表单业务动作优先使用 `isSubmitting`

### 5.2 方法命名

通用方法优先使用以下名称：

- `open`
- `closed`
- `getFormData`
- `getPayload`
- `fetchDetail`
- `submitForm`
- `submitAction`

不要使用过于随意的命名，例如：

- `initData`
- `ok`
- `doSubmit`
- `handleSure`

### 5.3 事件命名

弹窗对外事件优先统一为：

- `success`

仅在确实需要通知父组件额外行为时，再增加：

- `cancel`
- `opened`
- `closed`

默认不建议为了“点击取消”单独抛事件，除非父层需要联动。

## 6. Props 设计规范

### 6.1 新增 / 编辑类弹窗

推荐：

```ts
interface Props {
  isEdit: boolean;
  row: XxxItem | null;
}
```

说明：

- 仅有新增 / 编辑时，使用 `isEdit`
- 被编辑对象统一叫 `row`

### 6.2 多业务场景弹窗

如果弹窗不只是新增 / 编辑，而是多个业务动作共用一个弹窗，允许使用 `scene`：

```ts
type UserActionScene = "reset-password" | "assign-role" | "disable";

interface Props {
  scene: UserActionScene;
  row: UserItem | null;
}
```

说明：

- 当场景超过两种时，用 `scene`
- 不要继续滥用 `isEdit`
- `scene` 必须是明确的联合类型，不要直接写 `string`

## 7. 表单弹窗标准结构

推荐骨架：

```vue
<script setup lang="ts">
import { computed, reactive, ref, useTemplateRef } from "vue";
import { ElMessage, type FormInstance, type FormRules } from "element-plus";

interface Props {
  isEdit: boolean;
  row: XxxItem | null;
}

interface Emits {
  (e: "success"): void;
}

const props = defineProps<Props>();
const emit = defineEmits<Emits>();
const dialogVisible = defineModel<boolean>();
const formRef = useTemplateRef<FormInstance>("formRef");

const form = ref({
  name: "",
  remark: "",
});

const rules = reactive<FormRules<typeof form.value>>({});
const isSubmitLoading = ref(false);

const title = computed(() => (props.isEdit ? "编辑" : "新增"));

const open = () => {
  if (props.isEdit && props.row) {
    Object.assign(form.value, props.row);
  }
};

const closed = () => {
  formRef.value?.resetFields();
};

const getFormData = () => ({
  ...form.value,
});

const add = async () => {
  return await createXxxApi(getFormData());
};

const edit = async () => {
  return await updateXxxApi({
    id: props.row!.id,
    ...getFormData(),
  });
};

const submitForm = () => {
  if (!formRef.value) return;
  formRef.value.validate(async (valid) => {
    if (!valid) return;
    isSubmitLoading.value = true;
    try {
      const res = props.isEdit ? await edit() : await add();
      if (res.code === 0) {
        ElMessage.success(`${props.isEdit ? "编辑" : "新增"}成功`);
        emit("success");
        dialogVisible.value = false;
      }
    } finally {
      isSubmitLoading.value = false;
    }
  });
};
</script>
```

要求：

- 表单关闭时优先使用 `formRef.value?.resetFields()`
- 除非存在 `resetFields()` 无法覆盖的额外本地状态，否则不要额外引入 `createDefaultForm`
- 提交 loading 必须放在 `finally`
- 成功后顺序统一为：提示成功 → `emit("success")` → 关闭弹窗

## 8. 业务动作弹窗标准结构

业务动作弹窗不要求必须叫 `FormDialog`，但结构仍然保持一致。

推荐骨架：

```ts
interface Props {
  scene: "publish" | "offline";
  row: XxxItem | null;
}

const dialogVisible = defineModel<boolean>();
const isSubmitting = ref(false);

const open = () => {
  // 初始化上下文
};

const closed = () => {
  // 重置临时状态
};

const getPayload = () => ({
  id: props.row!.id,
  scene: props.scene,
});

const submitAction = async () => {
  isSubmitting.value = true;
  try {
    const res = await submitXxxActionApi(getPayload());
    if (res.code === 0) {
      ElMessage.success("操作成功");
      emit("success");
      dialogVisible.value = false;
    }
  } finally {
    isSubmitting.value = false;
  }
};
```

要求：

- 即使只有一个按钮，也要有清晰的 `getPayload`
- 不要把请求参数散落在 `submitAction` 里拼装
- 业务动作型弹窗也要有成功关闭与状态清理

## 9. 详情弹窗标准结构

推荐骨架：

```ts
const dialogVisible = defineModel<boolean>();
const loading = ref(false);
const detail = ref<XxxDetail | null>(null);

const open = async () => {
  if (!props.id) return;
  loading.value = true;
  try {
    detail.value = await fetchDetail();
  } finally {
    loading.value = false;
  }
};

const closed = () => {
  detail.value = null;
};
```

要求：

- 详情数据不要长期残留
- 如果打开就请求，统一放到 `open`
- 如果详情已经由父组件完整传入，可以不再请求，但仍要有 `closed` 清理

## 10. 页面与弹窗协作规范

页面层推荐只保留以下职责：

- `dialogVisible`
- `currentRow`
- `currentScene`
- `handleCreate`
- `handleEdit`
- `handleOpenActionDialog`
- `handleDialogSuccess`

不建议页面层直接做：

- 弹窗表单字段赋值
- 弹窗校验逻辑
- 弹窗内部接口拼装

推荐协作方式：

```vue
<BusinessActionDialog
  v-model="dialogVisible"
  :scene="currentScene"
  :row="currentRow"
  @success="handleDialogSuccess"
/>
```

## 11. 什么时候用 `isEdit`，什么时候用 `scene`

使用 `isEdit`：

- 只有新增 / 编辑两种状态
- 标题和接口只在 create / update 间切换

使用 `scene`：

- 超过两种状态
- 同一弹窗需要承载多个业务动作
- 不同动作对应不同标题、按钮文案、接口或字段

结论：

- CRUD 表单弹窗：优先 `isEdit`
- 业务动作弹窗：优先 `scene`

## 12. 项目落地约束

后续新增弹窗时，统一遵守以下规则：

1. 先判断弹窗类型：`FormDialog / ActionDialog / DetailDialog`
2. 只在“新增 / 编辑”两态下使用 `isEdit`
3. 多场景业务弹窗统一使用 `scene`
4. 统一使用 `defineModel<boolean>()`
5. 统一保留 `open / closed`
6. 统一通过 `success` 通知父组件刷新
7. 页面层不侵入弹窗内部表单实现

## 13. 与 CRUD 规范的关系

- `docs/CRUD页面规范.md` 是“标准管理页”规范
- 本文档是“所有弹窗组件”的上位规范

也就是说：

- CRUD 弹窗遵守本文档
- 非 CRUD 的业务弹窗也遵守本文档

如果两者出现重叠，以本文档的弹窗通用约束为准，再叠加 CRUD 页面特有约束。
