# 弹窗组件规范

## 1. 适用范围

适用于后台场景中的弹窗组件，包括：

- 新增/编辑类表单弹窗
- 业务确认类操作弹窗
- 只读详情弹窗

本文重点约束新增/编辑场景下 `FormDialog` 的职责边界，明确什么时候使用“业务弹窗”，什么时候拆成“通用表单 + 业务容器”。

## 2. 弹窗分类

### 2.1 BusinessFormDialog

指带有明确业务语义的新增/编辑弹窗，例如：

- `RoleFormDialog.vue`
- `UserFormDialog.vue`
- `PermissionFormDialog.vue`

职责：

- 管理表单状态
- 管理校验规则
- 处理编辑态回显
- 组装提交参数
- 直接调用新增/编辑接口
- 管理 `isSubmitLoading`
- 成功提示并触发 `success`

适用场景：

- 单一实体的标准 CRUD 新增/编辑
- 新增和编辑只是在字段、校验、提交参数上有轻微差异
- 表单不会被多个页面或多个提交流程复用

默认原则：

- 后台标准 CRUD 优先使用 `BusinessFormDialog`
- 不要为了“让父组件统一 submit”而提前抽象

### 2.2 CommonForm

指不直接承载业务提交的纯表单组件，例如：

- `RoleForm.vue`
- `UserForm.vue`

职责：

- 渲染字段
- 管理表单 model
- 管理校验规则
- 回显已有数据
- 暴露 `validate`、`getFormData`、`reset` 等能力

不负责：

- 直接调用接口
- 维护业务提交 loading
- 决定提交成功后的消息提示、刷新列表、关闭弹窗

适用场景：

- 同一套表单需要在弹窗、抽屉、页面内嵌区域复用
- 同一套字段会接入多个业务流程
- 新增、编辑、复制、另存为等场景的提交流程差异明显

### 2.3 ContainerDialog

指包裹 `CommonForm` 的业务容器弹窗。

职责：

- 控制弹窗显示/隐藏
- 接收父组件传入的 `submitting`
- 调用 `CommonForm` 暴露的校验与取值方法
- 决定提交时机
- 触发外层 `submit`

典型组合：

```text
页面 index.vue
  -> XxxFormDialog.vue
    -> XxxForm.vue
```

或：

```text
页面 index.vue
  -> 直接使用 XxxForm.vue
```

## 3. 选型原则

按以下顺序判断：

1. 如果只是标准 CRUD 新增/编辑，优先 `BusinessFormDialog`
2. 如果表单会被多个容器或多个流程复用，优先拆出 `CommonForm`
3. 如果拆出了 `CommonForm`，再根据业务决定是否需要 `ContainerDialog`

推荐判断标准：

- 只有一个页面会使用，且只有一个保存流程：用 `BusinessFormDialog`
- 同一表单需要“新增/编辑/复制”等多种提交编排：拆 `CommonForm`
- 父组件必须统一控制提交、loading、权限前置校验或二次确认：使用 `ContainerDialog`

不推荐：

- 没有明确复用需求时，把所有表单弹窗都改成父组件 submit
- 纯为抽象而抽象，导致父组件堆积表单细节

## 4. 职责边界

### 4.1 页面层

页面层负责：

- 查询条件
- 列表数据
- 选择态
- 打开/关闭弹窗
- 删除、批量删除
- 刷新列表

页面层在以下两种模式中的差异：

- 使用 `BusinessFormDialog` 时，只负责打开弹窗和监听 `success`
- 使用 `ContainerDialog` 时，负责传入 `submitting`、监听 `submit`、执行接口调用

### 4.2 BusinessFormDialog

建议统一命名：

- `dialogVisible`
- `formRef`
- `form`
- `rules`
- `isSubmitLoading`
- `isEdit`
- `open`
- `closed`
- `getFormData`
- `add`
- `edit`
- `submitForm`

成功流程建议：

1. 调用接口
2. 成功提示
3. 关闭弹窗
4. 触发 `success`

如果附近现有实现顺序不同，允许在同一模块内先保持一致；若要统一，需成组调整。

### 4.3 CommonForm

建议统一暴露能力：

- `validate`
- `getFormData`
- `reset`
- `setFormData`

建议入参：

- `modelValue` 或显式业务对象，如 `role`、`user`
- `isEdit`
- 纯展示所需的数据源，如 `roleOptions`

不建议：

- 在 `CommonForm` 内部直接调用 `createXxxApi`、`updateXxxApi`
- 在 `CommonForm` 内部持有业务提交态 `isSubmitLoading`

### 4.4 ContainerDialog

建议职责：

- 管理 `dialogVisible`
- 管理弹窗标题、按钮文案
- 接收 `submitting`
- 点击确定时调用表单 `validate`
- 通过 `emit('submit', formData)` 把数据交给父组件

建议命名：

- `submitting`
- `handleSubmit`
- `handleCancel`

## 5. 目录与命名建议

### 5.1 标准 CRUD 场景

目录示例：

```text
src/views/system/role/
  index.vue
  components/
    RoleFormDialog.vue
```

### 5.2 表单复用场景

目录示例：

```text
src/views/system/user/
  index.vue
  components/
    UserForm.vue
    UserFormDialog.vue
```

说明：

- `XxxForm.vue` 表示纯表单
- `XxxFormDialog.vue` 表示容器弹窗或业务弹窗
- 如果已经存在 `XxxFormDialog.vue`，不要无理由重命名

## 6. 当前仓库默认实现

当前仓库中的以下组件属于 `BusinessFormDialog`：

- `src/views/system/role/components/RoleFormDialog.vue`
- `src/views/system/user/components/UserFormDialog.vue`
- `src/views/system/permission/components/PermissionFormDialog.vue`
- `src/views/system/dictionary/components/DictionaryFormDialog.vue`

因此，当前默认实现方向为：

- CRUD 页面优先使用业务弹窗
- 出现明确复用需求后，再拆通用表单

## 7. 评审检查项

- 这是标准 CRUD 弹窗，还是复用型表单容器？
- 是否因为真实复用需求而拆分，而不是提前抽象？
- 页面层是否只保留页面职责，没有堆叠表单细节？
- `CommonForm` 是否保持纯表单职责，没有混入接口调用？
- `BusinessFormDialog` 是否保持提交逻辑聚合，没有把表单细节散落到页面？
- 命名、目录和事件语义是否清晰一致？
