# 07.接口和类型规范

## 1. 目标与范围

本规范用于统一前后端接口契约与 TypeScript 类型声明方式，核心目标：

- 降低跨团队沟通成本，避免“字段名一致但语义不一致”。
- 保证接口可演进（向后兼容），减少前端回归风险。
- 建立可维护的类型系统，减少 `any` 与运行时兜底逻辑。

适用范围：

- 后端：REST 接口设计、响应协议、错误码设计、版本兼容策略。
- 前端：`src/api/**`、`src/types/**`、`src/utils/http/**` 的接口声明与类型声明。

---

## 2. 统一接口契约（后端必须遵循）

### 2.1 响应包络（Envelope）

所有 JSON 业务接口统一返回：

```ts
interface ApiResponse<T> {
  code: number;
  data: T;
  message: string;
}
```

语义约定：

- `code = 0`：业务成功。
- `code != 0`：业务失败（如账号密码错误、权限不足）。
- `message`：可展示给用户的文案，要求可读。
- `data`：
  - 成功时为业务数据；
  - 失败时建议返回 `null` 或最小必要结构，避免返回不稳定字段。

对应前端实现：`src/types/http.ts`、`src/utils/http/interceptors.ts`。

### 2.2 HTTP 状态码与业务码职责划分

- HTTP 状态码：传输层/协议层语义（参数格式、认证、服务器异常）。
- 业务码 `code`：业务规则语义（登录失败、业务校验失败、权限拒绝）。

推荐实践：

- 参数格式错误：`400 + code=400xx`
- 未认证/令牌失效：`401 + code=401xx`
- 已认证但无权限：`403 + code=403xx`
- 业务失败但请求本身合法：`200 + code!=0`
- 服务端异常：`500 + code=500xx`

### 2.3 认证约定

- 认证头：`Authorization: Bearer <token>`
- 缺失或无效令牌时返回 `401`，并提供明确 `message`。
- 后端不得使用多种 token 头并存（避免前端分支逻辑）。

对应前端实现：`src/utils/http/interceptors.ts` 的 `attachAuthorization`。

### 2.4 路径与方法规范

- 路径使用名词，避免动词：`/auth/login`、`/auth/me`
- 方法语义：
  - `GET`：只读且幂等
  - `POST`：创建/登录/提交动作
  - `PUT/PATCH`：更新
  - `DELETE`：删除
- 路径采用小写+中划线或小写+斜杠分段，禁止驼峰路径。

### 2.5 版本与兼容策略

- 破坏性变更必须升版本（如 `/v2/...`）。
- 非破坏性演进优先通过“新增可选字段”完成。
- 禁止在不升版本情况下：
  - 字段重命名
  - 字段类型变更（如 `string -> number`）
  - 删除前端已使用字段

## 3. 前端接口声明规范（`src/api/**`）

### 3.1 目录与文件职责

- `src/api/<domain>/api.ts`：仅放接口函数，不放类型定义。
- `src/api/<domain>/types.ts`：仅放该域 DTO/VO 类型。
- `src/api/<domain>/index.ts`：统一导出，避免上层感知内部文件结构。

示例（当前项目）：

- `src/api/auth/api.ts`
- `src/api/auth/types.ts`
- `src/api/auth/index.ts`

### 3.2 接口函数声明规则

- 函数命名：`<action>Api`，如 `loginApi`、`getProfileApi`。
- 必须显式声明响应类型：`request<LoginResponse>(...)`。
- 请求体/查询参数必须有独立类型，不允许内联匿名对象。
- 不在 API 层做业务判断（如 `code===0`），只负责请求与返回契约。

推荐模板：

```ts
import { request } from "@/utils/http";
import type { XxxPayload, XxxResponse } from "./types";

export const createXxxApi = (payload: XxxPayload) => {
  return request<XxxResponse>("/xxx", "MOCK", {
    method: "post",
    data: payload,
  });
};
```

### 3.3 ServiceName 约束

- 第二参数必须使用 `ServiceName`，禁止硬编码 `baseURL`。
- 新增后端服务地址时，仅改 `src/utils/http/config.ts`。
- 业务 API 不直接感知网关地址，遵循依赖倒置原则（DIP）。

## 4. 接口类型声明规范（`types.ts` / `src/types/**`）

### 4.1 命名规范

- 请求体：`XxxPayload`
- 查询参数：`XxxQuery`
- 列表项：`XxxItem`
- 详情返回：`XxxDetail` 或 `XxxResponse`
- 分页结果：`PageResult<T>`
- 通用响应壳：`ApiResponse<T>`

### 4.2 `type` 与 `interface` 选型

- 对象结构优先 `interface`（可扩展、可声明合并）。
- 联合类型、映射类型、工具类型结果优先 `type`。
- 不混用同名 `type/interface`，避免认知噪音。

### 4.3 字段约束

- 禁止 `any`；未知结构使用 `unknown` 并在边界收窄。
- 可选字段必须是“真实可缺省”，不是“后端不确定”。
- 枚举语义优先字符串字面量联合，减少魔法数字。
- 时间字段明确格式：推荐 ISO 8601 字符串（如 `createdAt: string`）。
- ID 字段统一 `string`（与当前项目 `AppMenu.id`、`AuthUser.id` 保持一致）。

### 4.4 复用与去重（DRY）

- 通用类型放 `src/types/**`，业务域特有类型放 `src/api/<domain>/types.ts`。
- 跨模块复用类型从单一来源导出，避免重复定义同结构类型。
- 示例：`auth/types.ts` 直接复用并转导出 `menu` 相关类型。

### 4.5 类型导入导出规范

- 仅类型导入使用 `import type`（项目 ESLint 强制）。
- 统一使用 `index.ts` 作为模块出口，减少深层路径依赖。

## 5. 错误码分层建议（可直接落地）

建议按“领域 + 场景”分段，保持可读和可检索：

- `0`：成功
- `100xx`：认证域（如 `10001` 用户名或密码错误）
- `200xx`：用户域
- `300xx`：角色/权限域
- `400xx`：请求参数域（格式、必填、范围）
- `401xx`：登录态域（缺 token、token 过期）
- `500xx`：系统异常域

要求：

- `code` 全局唯一。
- `message` 面向终端用户，避免堆栈信息泄露。
- 错误码文档与实现必须同步更新。

## 6. 与当前项目对齐的最小示例

### 6.1 后端接口定义（契约）

- `POST /auth/login`
  - 请求：`LoginPayload`
  - 成功：`ApiResponse<{ accessToken: string }>`
- `GET /auth/me`
  - 请求头：`Authorization: Bearer <token>`
  - 成功：`ApiResponse<ProfileResponse>`

### 6.2 前端类型定义（摘录）

```ts
export interface LoginPayload {
  username: string;
  password: string;
}

export interface LoginResponse {
  accessToken: string;
}

export interface ProfileResponse {
  user: AuthUser;
  roles: string[];
  permissions: PermissionCode[];
  menus: AppMenu[];
}
```

## 7. 评审检查清单（PR 必查）

- 是否统一返回 `code/data/message`。
- 是否正确区分 HTTP 状态码与业务码。
- 是否新增了破坏性字段变更；若有，是否已升版本。
- 是否存在 `any`、匿名内联大对象、重复类型定义。
- 是否所有 API 都声明了明确泛型返回类型。
- 是否仅通过 `ServiceName` 选择服务，不在业务 API 写死地址。

---

## 8. 反模式（禁止）

- 在组件中直接写 Axios 请求，绕过 `src/api/**`。
- 后端返回不稳定结构（同一接口不同字段集）。
- 使用 `200 + code=0` 但 `message` 填错误文案。
- 将后端枚举值定义为 `number` 且不提供语义映射。
- 前端用 `as any` 强制吞掉类型错误。
