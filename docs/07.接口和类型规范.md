# 07.接口和类型规范

## 1. 目标与范围

本规范用于统一当前项目的接口契约和 TypeScript 类型声明方式，目标是：

- 保持前后端字段语义一致
- 降低类型漂移和运行时兜底逻辑
- 让接口扩展、Mock 联调和文档维护保持同步

适用范围：

- `src/api/**`
- `src/types/**`
- `src/utils/http/**`
- `mock/**`

## 2. 统一响应契约

当前项目统一使用如下业务响应包络：

```ts
interface ApiResponse<T> {
  code: number;
  data: T;
  message: string;
}
```

对应实现：

- `src/types/http.ts`
- `src/utils/http/client.ts`

语义约定：

- `code = 0`：业务成功
- `code != 0`：业务失败
- `message`：可读的业务提示
- `data`：成功时为业务数据，失败时建议返回 `null` 或最小必要结构

## 3. HTTP 状态码与业务码职责

- HTTP 状态码：网络层、协议层、认证层语义
- 业务码 `code`：业务结果语义

当前项目约定建议如下：

- 参数错误：`400 + 400xx`
- 未认证或 token 失效：`401 + 401xx`
- 已认证但无权限：`403 + 403xx`
- 请求成功但业务失败：`200 + code != 0`
- 服务端异常：`500 + 500xx`

## 4. 认证与服务地址规范

### 4.1 认证头

统一使用：

```http
Authorization: Bearer <token>
```

对应前端实现：`src/utils/http/client.ts`

### 4.2 服务地址

业务接口必须通过 `ServiceName` 选择服务地址，不能在 API 层写死 `baseURL`。

当前配置文件：`src/config/api/index.ts`

当前服务名：

- `MOCK`

新增服务时：

- 先在 `src/config/api/index.ts` 扩展 `SERVICE_URL_MAP`
- 再在业务 API 中按服务名引用

## 5. 前端接口声明规范

### 5.1 目录职责

- `src/api/<domain>/api.ts`：接口函数
- `src/api/<domain>/types.ts`：领域类型
- `src/api/<domain>/index.ts`：统一导出

当前示例：

- `src/api/auth/api.ts`
- `src/api/auth/types.ts`
- `src/api/auth/index.ts`

### 5.2 接口函数规则

- 函数名采用 `<action>Api`
- 必须显式声明返回泛型
- 请求体和响应体使用独立类型
- 不在 API 层做页面业务判断

示例：

```ts
import { request } from "@/utils/http";
import type { LoginPayload, LoginResponse } from "./types";

export const loginApi = (payload: LoginPayload) => {
  return request<LoginResponse>("/auth/login", "MOCK", {
    method: "post",
    data: payload,
  });
};
```

### 5.3 `request()` 配置项补充

当前 `request()` 支持以下项目内常用配置：

- `isPublic?: boolean`
  - 标记公开接口
  - 不自动携带 `Authorization`
  - 不参与 accessToken 失效后的自动刷新逻辑
- `skipErrorToast?: boolean`
  - 仅跳过 HTTP 层和业务层的统一错误提示
  - 不影响异常抛出、token 刷新、重放请求和登出清理

示例：

```ts
export const refreshTokenApi = (payload: RefreshTokenPayload) => {
  return request<RefreshTokenResponse>("/auth/refresh", "MOCK", {
    method: "post",
    data: payload,
    isPublic: true,
    skipErrorToast: true,
  });
};
```

使用建议：

- 登录、刷新 token、轮询探测等“调用方已有明确兜底处理”的接口，可按需开启 `skipErrorToast`
- 普通业务接口默认不要加，优先保持统一错误提示体验
- 不要把 `skipErrorToast` 当成“静默失败”开关；它只负责不弹统一错误提示
- 统一错误提示当前通过 `src/utils/toast.ts` 中的 `showErrorToast()` 实现

## 6. 当前认证域接口契约

基于现有代码，认证域最小契约如下：

```ts
interface LoginPayload {
  username: string;
  password: string;
}

interface LoginResponse {
  accessToken: string;
  refreshToken: string;
}

interface RefreshTokenPayload {
  refreshToken: string;
}

interface RefreshTokenResponse {
  accessToken: string;
  refreshToken: string;
}

interface CurrentUserResponse {
  user: AuthUser;
}

interface RolesResponse {
  roles: string[];
}

interface PermissionsResponse {
  permissions: PermissionCode[];
}

interface MenusResponse {
  menus: AppMenu[];
}
```

当前接口列表：

- `POST /auth/login`
- `POST /auth/refresh`
- `GET /auth/me`
- `GET /auth/roles`
- `GET /auth/permissions`
- `GET /auth/menus`

## 7. 类型声明规范

### 7.1 命名建议

- 请求体：`XxxPayload`
- 查询参数：`XxxQuery`
- 列表项：`XxxItem`
- 单接口返回：`XxxResponse`
- 通用响应壳：`ApiResponse<T>`

### 7.2 `type` 与 `interface`

- 对象结构优先 `interface`
- 联合类型、工具类型、字面量组合优先 `type`

### 7.3 字段约束

- 禁止使用 `any`
- 未知结构优先 `unknown`
- 时间字段明确格式
- ID 字段在当前项目中统一为 `string`
- 仅类型导入时必须使用 `import type`

## 8. 菜单与路由类型约定

当前菜单相关类型位于 `src/types/menu.ts`：

```ts
export interface AppRouteMeta {
  title: string;
  icon?: string;
  rank?: number;
  hidden?: boolean;
  keepAlive?: boolean;
  permission?: PermissionCode | PermissionCode[];
  public?: boolean;
}

export interface AppMenu {
  id: string;
  name: string;
  path: string;
  component?: string;
  redirect?: string;
  meta: AppRouteMeta;
  children?: AppMenu[];
}
```

约束建议：

- `name` 必须稳定且唯一
- `path` 用于路由拼接和菜单索引，建议保持语义化
- 后端菜单子节点 `path` 可使用完整路径，如 `system/user`；前端会兼容相对路径与完整路径
- `component` 采用相对 `src/views` 的路径，不带扩展名
- `meta.permission` 与后端权限码保持一致

## 9. 下载接口补充约定

当前 `request()` 已支持下载场景：

- `responseType: "blob"`
- `responseType: "arraybuffer"`

相关类型定义位于：

- `src/utils/http/types.ts`

如果新增下载接口：

- 要明确 `responseType`
- 不再假定返回 `ApiResponse<T>`
- 调用方自行按下载数据类型处理结果

## 10. 维护建议

- 接口变更时，同时更新类型、Mock、文档和页面调用
- 新增权限相关接口时，优先保持菜单、权限、角色数据一致
- 若发生破坏性字段变更，避免只改页面而不改中间层类型
- 评审时优先检查是否存在深层重复类型、匿名对象和 `as any`
