# 06.代码规范

## 1. 目标与范围

本项目使用 `ESLint + Prettier + TypeScript + EditorConfig` 作为统一规范方案，目标是：

- 减少多人协作时的风格漂移
- 尽早发现语法、类型和常见逻辑问题
- 让提交和评审更聚焦于真实业务改动

适用范围：

- `src/`
- `mock/`
- `docs/`
- 根目录配置文件中的 `js/ts/vue/md/json/cjs` 等文件

## 2. 执行命令

```bash
pnpm run typecheck # 执行 TypeScript 类型检查
pnpm run lint # 执行 ESLint 检查
pnpm run lint:fix # 自动修复可处理的 ESLint 问题
pnpm run format # 使用 Prettier 格式化项目文件
pnpm run format:check # 检查项目文件是否符合 Prettier 规范
pnpm run lint:style # 执行 Stylelint 检查
pnpm run check # 执行综合检查（typecheck + lint + lint:style）
pnpm run prepare # 安装或刷新 Husky Git hooks
```

说明：

- `lint` 使用 `--max-warnings=0`
- `check` 当前等于 `typecheck + lint + lint:style`
- `format:check` 没有并入 `check`，需要单独执行

## 3. Git Hooks（Husky）

项目已接入 Husky。

当前相关配置：

- `package.json`：`prepare: "husky"`
- Hook 文件：`.husky/pre-commit`

`pre-commit` 当前执行：

```bash
pnpm run check # 提交前执行类型和 Lint 综合检查
pnpm run format:check # 提交前检查格式是否符合 Prettier 规范
```

执行策略：

- 任一步骤失败都会阻止提交
- 不建议跳过 Hook；如确需跳过，应在提交说明里明确原因

## 4. ESLint 规范

配置文件：`eslint.config.mjs`

### 4.1 基础配置

- 使用 ESLint Flat Config
- 组合规则：
  - `@eslint/js`
  - `typescript-eslint`
  - `eslint-plugin-vue`
  - `eslint-config-prettier`
- 语言环境：
  - `ecmaVersion: "latest"`
  - 默认 `sourceType: "module"`
  - `*.cjs` 例外为 `commonjs`
  - 全局变量：`browser + node`

### 4.2 忽略路径

- `dist/**`
- `node_modules/**`
- `.pnpm-store/**`
- `coverage/**`
- `*.min.js`

### 4.3 关键规则

错误级别：

- `curly: ["error", "all"]`
- `eqeqeq: ["error", "always"]`
- `no-debugger: "error"`
- `@typescript-eslint/consistent-type-imports: "error"`
- `@typescript-eslint/no-unused-vars: "error"`
- `vue/block-order: "error"`
- `vue/padding-line-between-blocks: ["error", "always"]`
- `vue/multi-word-component-names: "error"`

补充说明：

- `_` 前缀的未使用参数、变量、异常参数允许保留
- `vue/multi-word-component-names` 当前仅忽略 `Index`

警告级别：

- `no-console: "warn"`，只允许 `console.warn`、`console.error`
- `vue/no-v-html: "warn"`

### 4.4 项目级例外

为兼容当前图标实现，以下文件做了定点例外：

- 文件：`src/components/ReIcon/src/hooks.ts`
- 规则：`vue/one-component-per-file = off`

除该文件外，不建议继续扩大豁免范围。

## 5. Prettier 规范

配置文件：`prettier.config.mjs`

当前主要参数：

- `printWidth: 100`
- `tabWidth: 2`
- `useTabs: false`
- `semi: true`
- `singleQuote: false`
- `trailingComma: "all"`

## 6. EditorConfig 规范

配置文件：`.editorconfig`

统一基础行为：

- `charset = utf-8`
- `end_of_line = lf`
- `indent_style = space`
- `indent_size = 2`
- `insert_final_newline = true`
- `trim_trailing_whitespace = true`

特殊规则：

- Markdown 文件保留行尾空格：`[*.md] trim_trailing_whitespace = false`

## 7. Vue 与工具约定

### 7.1 组件内书写顺序

`<script setup>` 内建议按以下顺序组织：

1. `import`
2. `defineOptions`
3. `defineProps` / `defineEmits` / `defineModel`
4. `onMounted`
5. 常量、类型补充
6. `ref` / `reactive` / `computed`
7. 业务方法
8. `watch` / `watchEffect`
9. `defineExpose`

约定：

- `onMounted` 默认放在 `defineProps` / `defineEmits` / `defineModel` 之后，作为组件入口
- `onMounted` 内优先只放初始化调度逻辑，例如 `getRoleList()`、`initData()`、`loadOptions()`
- 页面层列表加载方法优先使用 `getXxxList()` 命名，避免使用语义过泛的 `fetchList()`
- 布尔状态优先使用 `isXxx` 命名；页面列表加载态优先使用 `isListLoading`，提交态优先使用 `isSubmitLoading`
- 页面如果需要暂存当前编辑项，优先使用 `currentItem` 或 `currentRole`、`currentUser` 这类领域化命名；弹窗 props 避免使用语义过泛的 `row`
- 不建议在 `onMounted` 中直接堆叠大段业务实现，复杂逻辑应下沉到独立方法

### 7.2 公共提示与校验

- 统一提示能力优先收敛到 `src/utils/toast.ts`
- HTTP 层统一错误提示复用 `showErrorToast()`
- 可复用的表单正则、校验函数优先沉淀到 `src/utils/validators.ts`
- `validators.ts` 内建议按“正则紧跟对应校验函数”的顺序组织，便于维护

### 7.3 FormDialog 约定

- 新增/编辑类弹窗统一使用 `defineModel<boolean>()` 配合 `open`、`closed` 生命周期方法
- 标准 CRUD 场景默认使用 `BusinessFormDialog`，由弹窗内部负责表单、提交和 `isSubmitLoading`
- 只有当同一套表单需要被多个容器或多个业务流程复用时，才拆为 `CommonForm` 或 `CommonForm + ContainerDialog`
- 编辑态表单回显优先复用表单字段循环赋值，并将这段逻辑提炼为 `assignFormFromRole`、`assignFormFromUser` 这类小方法，避免在 `open` 中直接堆叠实现细节
- 只有当表单字段与接口字段明显不一致，或存在额外转换逻辑时，再改为显式逐字段映射
- 不要为了抽象而把所有表单弹窗都改为父组件 submit；页面层只在确有复用或编排需求时接管提交
- 详细分类、职责边界与选型规则见 `docs/08-弹窗组件规范.md`

### 7.4 CRUD 页面布局约定

- CRUD 页面外层优先使用 `no-bg-content`，不要再额外包整页白底
- 查询区与内容区统一拆分为两个区域，并复用全局卡片类 `app-card`
- `app-card` 负责统一圆角、边框、背景和内边距，不在页面里重复拼接同类 Tailwind 样式
- 查询区中的 `查询`、`重置` 归属于筛选操作，必须放在查询区内部，顺序为“查询在前，重置在后”
- 查询按钮组默认作为行尾操作，桌面端右对齐；不要挪到内容区工具栏
- 内容区顶部只放列表级操作，例如 `新建`、`批量删除`
- 单行查询表单需要去掉 `el-form-item` 默认的 `margin-bottom`，避免卡片底部留白过大
- 详细示例与页面拆分规范见 `docs/CRUD页面规范.md`

## 8. 团队协作建议

建议本地开发流程：

1. 开发完成后执行 `pnpm run format`
2. 再执行 `pnpm run check`
3. 提交前执行 `pnpm run format:check`

评审时重点关注：

- 是否通过类型和 Lint 校验
- 是否新增了不必要的规则例外
- 是否出现 `any`、无意义断言或深层路径硬编码
- 文档、Mock、类型和实现是否同步更新
