# 06.代码规范

## 1. 目标与范围

本项目采用 `ESLint + Prettier + TypeScript` 作为统一代码规范方案，目标是：

- 统一多人协作下的代码风格，减少无意义代码差异。
- 在提交前尽早发现语法、类型与常见逻辑问题。
- 将“风格问题”和“业务问题”分离，降低 Review 成本。

适用范围：

- `src/`、`mock/`、`docs/`、根目录配置文件中的 `js/ts/vue/md/json/cjs` 等文件。

## 2. 执行命令（以项目根目录为准）

```bash
pnpm run typecheck      # TypeScript 类型检查（vue-tsc）
pnpm run lint           # ESLint 检查（0 warning 门禁）
pnpm run lint:fix       # ESLint 自动修复
pnpm run format         # Prettier 全量格式化
pnpm run format:check   # Prettier 格式校验
pnpm run check          # typecheck + lint（当前合并门禁）
pnpm run prepare        # 安装/刷新 Husky Git hooks
```

说明：

- `lint` 使用 `--max-warnings=0`，任何 warning 都会导致命令失败。
- `check` 当前不包含 `format:check`，格式校验通过 `format:check` 独立执行。

## 3. Git Hooks（Husky）

项目已接入 Husky，用于在 `pre-commit` 阶段执行质量门禁。

- 依赖：`husky`
- 脚本：`package.json` 中 `prepare: "husky"`
- Hook 文件：`.husky/pre-commit`

`pre-commit` 当前执行：

```bash
pnpm run check
pnpm run format:check
```

执行策略：

- 任一步骤失败都会阻止提交。
- 不建议跳过 Hook；如确需跳过，必须在 PR 说明原因并补做校验。

## 4. ESLint 规范（eslint.config.mjs）

### 4.1 基础配置

- 使用 ESLint Flat Config（`eslint.config.mjs`）。
- 组合规则：
  - `@eslint/js`（JS 推荐规则）
  - `typescript-eslint`（TS 推荐规则）
  - `eslint-plugin-vue`（Vue 推荐规则）
  - `eslint-config-prettier`（关闭与 Prettier 冲突规则）
- 语言环境：
  - `ecmaVersion: "latest"`
  - `sourceType: "module"`（`*.cjs` 例外为 `commonjs`）
  - 全局变量：`browser + node`

### 4.2 忽略路径

以下路径不参与 ESLint：

- `dist/**`
- `node_modules/**`
- `.pnpm-store/**`
- `coverage/**`
- `*.min.js`

### 4.3 关键规则

错误级别（必须修复）：

- `curly: ["error", "all"]`：所有分支必须使用花括号。
- `eqeqeq: ["error", "always"]`：禁止 `==`，必须使用 `===`。
- `no-debugger: "error"`：禁止调试断点。
- `@typescript-eslint/consistent-type-imports: "error"`：统一使用 `import type`。
- `@typescript-eslint/no-unused-vars: "error"`：
  - 允许 `_` 前缀的未使用参数/变量/异常参数。
- `vue/block-order: "error"`：Vue SFC 块顺序固定为 `template -> script -> style`。
- `vue/padding-line-between-blocks: ["error", "always"]`：SFC 各块之间必须保留空行。

警告级别（建议修复）：

- `no-console: "warn"`：仅允许 `console.warn`、`console.error`。
- `vue/no-v-html: "warn"`：谨慎使用 `v-html`，避免 XSS 风险。

### 4.4 项目级例外

为兼容现有图标渲染实现，以下规则做了定点豁免：

- 文件：`src/components/ReIcon/src/hooks.ts`
- 规则：`vue/one-component-per-file = off`

要求：除该文件外，不得新增同类全局豁免；若需新增，必须在 PR 说明原因。

## 5. Prettier 规范（prettier.config.mjs）

主要格式化参数：

- `printWidth: 100`
- `tabWidth: 2`
- `useTabs: false`
- `semi: true`
- `singleQuote: false`
- `trailingComma: "all"`
- `arrowParens: "always"`
- `endOfLine: "lf"`
- `vueIndentScriptAndStyle: false`

插件：

- `prettier-plugin-tailwindcss`（自动排序 Tailwind 类名）

忽略文件（`.prettierignore`）：

- `dist`
- `node_modules`
- `.pnpm-store`
- `coverage`
- `pnpm-lock.yaml`

## 6. EditorConfig 规范（.editorconfig）

统一编辑器基础行为：

- `charset = utf-8`
- `end_of_line = lf`
- `indent_style = space`
- `indent_size = 2`
- `insert_final_newline = true`
- `trim_trailing_whitespace = true`
- Markdown 文件保留行尾空格（`[*.md] trim_trailing_whitespace = false`）

## 7. 团队协作约定

建议本地开发流程：

1. 开发完成后先执行 `pnpm run format`。
2. 再执行 `pnpm run check`。
3. 提交前执行 `pnpm run format:check`，确保无格式漂移。

PR 评审约束：

- 不接受关闭全局规则来绕过问题。
- 规则例外必须“定点、可解释、可追溯”。
- 规范升级（新增/调严规则）需在文档和配置同步更新。
