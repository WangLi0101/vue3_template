# 路由使用

## 当前路由结构

项目使用 `vue-router 4`，历史模式为 `createWebHashHistory()`。

当前路由由三部分组成：

- 静态路由：`src/router/static-routes.ts`
- 模块路由：`src/router/modules/**/*.ts`
- 动态菜单路由工具：`src/router/dynamic-routes.ts`

最终都归属于 `Root` 布局路由：

- 静态路由直接注册到 router
- 模块路由在创建 `Root.children` 时注入
- 当前业务页面由前端模块路由统一声明
- 动态菜单路由工具保留在仓库中，当前默认不参与运行时挂载

## 1. Router 入口

入口文件：`src/router/index.ts`

职责：

- 创建路由实例
- 注册静态路由
- 提供全局滚动行为

```ts
const router = createRouter({
  history: createWebHashHistory(),
  routes: staticRoutes,
  scrollBehavior() {
    return { top: 0 };
  },
});
```

## 2. 静态路由

定义文件：`src/router/static-routes.ts`

当前固定存在的路由：

- `Login`：`/login`，公开路由，`meta.public = true`
- `Forbidden`：`/403`，受保护路由
- `Root`：`/`，后台主布局
- `NotFound`：`/:pathMatch(.*)*`

其中 `Root.children` 初始只包含模块路由：

```ts
{
  path: "/",
  name: "Root",
  component: AppLayout,
  children: [...rootRouteModules],
}
```

## 3. 模块路由

目录：`src/router/modules`

聚合文件：`src/router/module-routes.ts`

实现特点：

- 使用 `import.meta.glob("./modules/**/*.ts", { eager: true })` 自动收集
- 支持 `default` 和 `routes` 两种导出形式
- 按 `meta.rank` 升序排序
- 在应用启动阶段就已注册，不依赖登录接口

当前仓库中的 `src/router/modules/nested.ts` 演示了三级路由和 `AppRouterView` 容器写法。

推荐写法：

```ts
import type { RouteRecordRaw } from "vue-router";
import AppRouterView from "@/components/AppRouterView.vue";

export const routes: RouteRecordRaw[] = [
  {
    path: "report",
    name: "Report",
    component: AppRouterView,
    meta: {
      title: "报表中心",
      rank: 10,
    },
    children: [
      {
        path: "list",
        name: "ReportList",
        component: () => import("@/views/report/ReportListPage.vue"),
        meta: {
          title: "报表列表",
        },
      },
    ],
  },
];
```

## 4. 动态菜单路由工具

`src/router/dynamic-routes.ts` 仍然保留，用于兼容后续如果需要恢复“后端菜单转动态路由”的场景。

当前默认链路里：

1. `authStore.initializeSession()` 仍会拉取菜单数据
2. `menuStore.setSidebarBackendMenus(menus)` 保存后端返回的左侧菜单树
3. `sidebarMenus` 直接消费后端菜单
4. `menuStore.dynamicRoutes` 仍然保留，但默认不写入运行时数据
5. 页面路由本身由 `src/router/modules/**/*.ts` 静态提供

如果未来恢复动态路由模式，`buildRoutesFromMenus()` 的转换规则仍然有效：

- 菜单按 `meta.rank` 排序
- `component === "router-view"` 时使用 `AppRouterView`
- 有 `children` 但未配置 `component` 时自动回退到 `AppRouterView`
- 未显式声明 `redirect` 且存在子节点时，自动跳到第一个可见子菜单
- 组件路径无效时降级为 `NotFoundPage`，并输出 `[RBAC]` 警告日志

菜单组件路径映射规则：

- 后端返回 `dashboard/DashboardPage`
- 前端最终会匹配到 `src/views/dashboard/DashboardPage.vue`

## 5. 路由守卫

定义文件：`src/router/guard.ts`

当前守卫职责：

- 处理公开路由放行
- 未登录时跳转登录页并带上 `redirect`
- 首次进入受保护页面时初始化会话
- 首次初始化完成后仍保留动态路由挂载入口
- 访问根路径 `/` 时跳转到当前首页菜单
- 基于 `meta.permission` 做页面级权限校验
- 使用 `NProgress` 处理切换进度条

守卫逻辑摘要：

1. 公开路由直接放行；已登录再访问登录页时跳回 `/`
2. 无 token 时跳转 `/login?redirect=目标地址`
3. 有 token 但尚未初始化时：
   - 并行拉取用户、角色、权限、菜单
   - 调用动态路由挂载逻辑，但默认不会挂入新路由
   - 若目标为 `/`，重定向到首页菜单
   - 否则 `replace` 回当前目标地址，使用前端已注册的静态路由继续匹配
4. 已初始化时：
   - 若访问 `/`，继续做首页重定向
   - 若存在 `meta.permission`，使用 `permissionStore.hasAll()` 校验
   - 校验失败跳转 `/403`

说明：

- 页面级权限是 AND 语义
- 公开路由依赖 `meta.public`
- 未声明 `meta.public` 的路由都会进入登录校验流程

## 6. 菜单、侧边栏与面包屑关系

相关实现：`src/stores/modules/menu.ts`、`src/utils/menu.ts`

`menuStore` 当前维护三类结果：

- `backendMenus`：预留给动态路由转换的后端菜单树，当前默认空数组
- `sidebarBackendMenus`：后端返回的左侧菜单树
- `allMenus`：动态菜单与前端模块路由合并后的完整菜单树
- `sidebarMenus`：后端菜单转换出的侧边栏树

当前规则：

- 前端模块路由负责完整页面路由和面包屑索引
- 后端菜单只负责控制左侧导航显示哪些入口
- 动态路由相关代码和挂载入口继续保留，当前默认不写入菜单数据
- `allMenus` 会合并 `backendMenus` 和前端静态路由菜单，确保后续恢复动态路由时面包屑和菜单索引仍然完整
- `sidebarMenus` 会继续过滤 `hidden` 节点

这意味着：

- 前端可以维护完整路由树，不依赖后端补齐隐藏页或详情页
- 后端只需要返回当前用户左侧可见菜单
- 动态路由未来一旦重新给 `backendMenus` 赋值，面包屑仍然可以从完整菜单树和 `matched` 链路中恢复

面包屑规则：

- 优先使用当前路由的 `matched` 链路，兼容隐藏页和详情页
- 若 `matched` 链路不可用，再回退到菜单索引查找
- 始终补齐“首页”节点

## 7. 根路径重定向

当用户访问 `/` 时，不会固定跳到硬编码首页，而是走 `menuStore.resolveRootRedirectTarget("/")`。

规则：

- 从当前 `sidebarMenus` 取第一个可见菜单
- 若首个菜单路径有效且不为 `/`，则重定向到该路径
- 若没有可用菜单，则不做重定向

当前首页标签和首页面包屑也围绕这个“第一个可见菜单”同步。

## 8. KeepAlive 与 RouterView 约定

承接组件：`src/components/AppRouterView.vue`

当前行为：

- `route.meta.keepAlive = true` 时，页面包裹在 `<KeepAlive>` 中
- 缓存 key 使用 `route.name || route.path`
- 非缓存页面使用 `route.fullPath` 作为 key
- 所有页面统一走 `route-switch` 过渡动画

如果某个父级节点只是路由容器，建议使用以下任一方式：

- 在静态模块路由里显式使用 `AppRouterView`
- 在后端菜单中声明 `component: "router-view"`

## 9. 新增路由的建议方式

新增固定功能页：

- 放到 `src/router/modules`
- 适用于框架内置页面、演示页、无需后端控制显示与否的页面

新增受后端控制的业务页：

- 由后端返回 `menus`
- 前端负责补齐 `src/views` 组件并保证 `component` 路径可解析

选择建议：

- “是否在导航中出现”需要后端控制时，用动态菜单路由
- “模板内置能力或固定示例”用模块路由

## 10. 退出登录后的状态清理

调用 `permissionStore.reset()` 与 `menuStore.reset()` 时会：

- 清空角色和权限集合
- 清空后端返回的菜单树
- 让首页、侧边栏和标签页在下次登录时重新初始化
