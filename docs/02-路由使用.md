# 路由使用

## 当前路由结构

项目使用 `vue-router 4`，历史模式为 `createWebHashHistory()`。

当前路由由三部分组成：

- 静态路由：`src/router/static-routes.ts`
- 模块路由：`src/router/modules/**/*.ts`
- 动态菜单路由：`src/router/dynamic-routes.ts`

最终都挂载在 `Root` 布局路由下：

- 静态路由直接在初始化时注册
- 模块路由在创建 router 时作为 `Root.children` 注入
- 动态菜单路由在登录后首次进入受保护页面时，通过 `router.addRoute("Root", route)` 注入

## 1. Router 入口

入口文件：`src/router/index.ts`

职责：

- 创建 router 实例
- 注册 `staticRoutes`
- 配置全局滚动行为：切换路由后回到顶部

```ts
const router = createRouter({
  history: createWebHashHistory(),
  routes: staticRoutes,
  scrollBehavior() {
    return { top: 0 };
  },
});
```

## 2. 静态路由

定义文件：`src/router/static-routes.ts`

当前固定存在的路由：

- `Login`：`/login`，公开路由，`meta.public = true`
- `Forbidden`：`/403`
- `Root`：`/`，后台主布局，组件为 `AppLayout`
- `NotFound`：`/:pathMatch(.*)*`

其中 `Root.children` 初始只包含模块路由：

```ts
{
  path: "/",
  name: "Root",
  component: AppLayout,
  children: [...rootRouteModules],
}
```

## 3. 模块路由

目录：`src/router/modules`

聚合文件：`src/router/module-routes.ts`

实现特点：

- 使用 `import.meta.glob("./modules/**/*.ts", { eager: true })` 自动收集
- 支持 `default` 和 `routes` 两种导出形式
- 按 `meta.rank` 升序排序，无 `rank` 时排在末尾
- 在应用启动时就已注册，不依赖登录接口返回

示例：

```ts
import type { RouteRecordRaw } from "vue-router";

export const routes: RouteRecordRaw[] = [
  {
    path: "demo",
    name: "DemoPage",
    component: () => import("@/views/dashboard/DashboardPage.vue"),
    meta: {
      title: "模块示例",
      requiresAuth: true,
      rank: 1,
    },
  },
];
```

当前仓库中的 `src/router/modules/nested.ts` 就是模块路由示例，演示了三级路由结构。

## 4. 动态菜单路由

动态路由来源于 `GET /api/auth/me` 返回的 `menus`，由 `menuStore.dynamicRoutes` 计算得到。

核心链路：

1. `authStore.fetchProfile()` 拉取 `menus`
2. `menuStore.setMenus(profile.menus)` 保存原始菜单
3. `buildRoutesFromMenus(rawMenus)` 转换为 `RouteRecordRaw[]`
4. `permissionStore.mountDynamicRoutes(router, menuStore.dynamicRoutes)` 注入到 `Root`

转换规则：

- 菜单按 `meta.rank` 排序
- `menu.component === "router-view"` 时使用 `AppRouterView`
- 有 `children` 但未配置 `component` 时，也自动回退到 `AppRouterView`
- 节点未配置 `redirect` 且有子路由时，自动重定向到首个未隐藏子节点
- 组件路径无效时，降级到 `NotFoundPage`，并输出 `[RBAC]` 警告日志

## 5. 路由守卫

定义文件：`src/router/guard.ts`

当前守卫职责：

- 处理公开路由放行
- 未登录跳转登录页，并带上 `redirect`
- 首次进入受保护页面时完成权限初始化
- 根路径 `/` 自动重定向到首个可访问菜单
- 基于 `meta.permission` 做页面级权限校验
- 配合 `NProgress` 处理路由切换进度条

守卫逻辑摘要：

1. 如果目标路由是公开路由，则直接放行。
2. 如果访问 `Login` 且已登录，则重定向到 `/`。
3. 如果没有 token，则跳转 `/login?redirect=目标地址`。
4. 如果有 token 但尚未初始化：
   - 调用 `authStore.fetchProfile()`
   - 注入动态路由
   - 必要时处理根路径重定向
   - 重新进入当前目标地址
5. 如果路由声明了 `meta.permission`，则使用 `permissionStore.hasAll()` 校验，失败跳转 `/403`。

说明：

- 当前页面级权限是“全部命中”语义。
- 公开路由依赖 `meta.public`，不是依赖 `requiresAuth = false`。

## 6. 菜单、路由与面包屑关系

相关实现：`src/stores/modules/menu.ts`

`menu store` 维护三类衍生结果：

- `sidebarMenus`：侧边栏菜单
- `dynamicRoutes`：后端菜单转换后的动态路由
- `getBreadcrumbs(path)`：面包屑链路

当前侧边栏并不是只展示后端菜单，而是：

- 先将后端 `menus` 转成侧边栏结构
- 再将 `rootRouteModules` 转成侧边栏结构
- 按 `path` 去重后合并

这意味着：

- 模块路由可以独立出现在侧边栏中
- 后端菜单与模块路由的同路径节点，后者会被去重忽略

## 7. 根路径重定向

当用户访问 `/` 时，不会固定跳到某个硬编码页面，而是走 `menuStore.resolveRootRedirectTarget()`。

规则：

- 取当前侧边栏中的第一个可见菜单
- 如果该菜单已有记录过的 `fullPath`，优先跳转 `fullPath`
- 否则跳转菜单本身的 `path`

`AppHeader` 会在路由变化时调用 `menuStore.rememberRoute(route.path, route.fullPath)`，用于记录带参数的真实访问地址。

## 8. KeepAlive 与 RouterView 约定

动态菜单和模块路由都可以通过 `meta.keepAlive` 控制缓存。

承接组件：`src/components/AppRouterView.vue`

当前行为：

- `route.meta.keepAlive = true` 时，页面包裹在 `<KeepAlive>` 中
- 缓存 key 使用 `route.name || route.path`
- 非缓存页面使用 `route.fullPath` 作为 key，每次完整路径变化都会重新渲染

如果某个父级节点只是路由容器，应使用 `AppRouterView` 或在菜单中声明 `component: "router-view"`。

## 9. 新增路由的建议方式

新增固定功能页：

- 放到 `src/router/modules`
- 适用于不依赖后端菜单返回、希望开箱即用的页面

新增受后端控制的业务页：

- 由后端返回 `menus`
- 前端只负责组件映射和动态注入

选择原则：

- “页面是否出现”由后端控制，用动态菜单路由
- “框架内置示例/固定页面”用模块路由

## 10. 退出登录后的路由清理

`permissionStore` 会保存每个动态路由的移除回调。

调用 `permissionStore.reset()` 时会：

- 清空角色与权限
- 执行所有 `removeRoute` 回调
- 重置 `isRoutesMounted`

因此退出登录后，动态菜单路由会被真实卸载，而不是只清空菜单状态。
